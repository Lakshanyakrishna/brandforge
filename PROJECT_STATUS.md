_Generated by code audit on 2026-07-04. No code was changed to produce this report._

## 1. Project Overview

**What it actually does today:** This is a backend-only Node/Express API (no frontend exists in the repo, despite what the README implies). A user registers/logs in with a `username` + `password` (JWT issued and set as a plain cookie). Once authenticated, they can `POST` a single image to a protected endpoint. The server converts the image to base64, sends it to Google Gemini (`gemini-2.5-flash`) with a fixed prompt ("short creative caption under 10 words with emojis and hashtags"), uploads the original image buffer to ImageKit, and (attempts to) save a `Post` document linking the caption, image URL, and user. The API responds with the caption and the ImageKit URL.

That's the entire functional surface: **register → login → upload one image → get an AI caption + hosted image URL back.** There is no endpoint to list, fetch, or delete posts, no logout route, and no user-profile route (one is stubbed as a comment only).

## 2. Tech Stack

| Layer | Actual |
|---|---|
| Backend | Node.js + **Express 5.2.1** |
| Database | **MongoDB** via **Mongoose 9.1.3** |
| Frontend | **None exists.** README describes a React + Tailwind `client/` folder; it is not present anywhere in the repo. |
| Auth | JWT (`jsonwebtoken 9.0.3`) + `bcryptjs 3.0.3`, token carried in a cookie via `cookie-parser` |
| AI / LLM | **Google Gemini**, via the `@google/generative-ai` SDK (`genAI.getGenerativeModel({model: "gemini-2.5-flash"})`), called from `src/services/ai.service.js`. Vision input (base64 image) + text prompt → caption text output. |
| Image storage | **ImageKit**, via the `imagekit` SDK, called from `src/services/storage.service.js` |
| File upload | `multer` (in-memory storage) |
| Misc | `uuid` (filename generation), `dotenv`, `nodemon` |

### Dependency flags

- **Dead/duplicate dependency:** `@google/genai` (the new unified Google GenAI SDK) is installed but **never imported anywhere**. The code exclusively uses the older `@google/generative-ai` SDK, which Google has been steering users away from in favor of `@google/genai`. Worth consolidating on one.
- **Dead/duplicate dependency:** `@imagekit/nodejs` (v7.2.0) is installed but unused — the code uses the older `imagekit` package (v6.0.0) instead.
- `nodemon` is listed under `dependencies` instead of `devDependencies` — harmless but sloppy; it ships in production installs for no reason.
- No test framework of any kind. `package.json`'s `test` script is the default create-react/npm placeholder that just exits with an error.
- `mongoose ^9.1.3` is a notably new major version — worth double-checking release notes/breaking changes if you're used to the more commonly documented v8.x API.
- No `cors` package anywhere — if you ever attach a browser-based frontend on a different origin, requests will be blocked by CORS with the current `app.js`.


There is no `public/`, `views/`, `client/`, or static-serving anywhere — confirming zero frontend code exists.

### Entry points
- **Server start:** `server.js` — hardcodes port `3000` (the README's `.env` example sets `PORT=5000`, but nothing in the code ever reads `process.env.PORT`, so that variable currently does nothing).
- **App wiring:** `src/app.js`.
- **No frontend root** — none exists.

### API Endpoints

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/api/auth/register` | none | Create user (`username`, `password`), hash password, set JWT cookie |
| POST | `/api/auth/login` | none | Validate credentials, set JWT cookie |
| POST | `/api/posts/` | **required** (cookie JWT) | Upload one image (`multipart/form-data`, field `image`) → Gemini caption → ImageKit upload → save Post → return `{post, caption, imageUrl}` |

No GET/DELETE/PUT routes exist anywhere in the app.

### Database Schema

**User** (`src/models/user.model.js`)
```js
{ username: String (unique, required), password: String (bcrypt hash) }
```

**Post** (`src/models/post.model.js`)
```js
{ image: String, caption: String, user: ObjectId (ref: 'User') }
```

**Bug found:** the schema field is named `image`, but `post.controller.js` calls `postModel.create({ caption, imageUrl: result.url, user })` — it writes to a key `imageUrl` that **does not exist in the schema**. Under Mongoose's default `strict: true` mode, unknown fields are silently dropped, meaning **the image URL is returned in the API response but never actually persisted to MongoDB.** The `image` field is always empty in the database.

**Latent bug:** `User` is registered as `mongoose.model('user', ...)` (lowercase `'user'`), but `Post.user` declares `ref: 'User'` (capital `U`). Mongoose model names are case-sensitive, so `.populate('user')` on a Post would currently fail to resolve — not an active bug since nothing calls `.populate()` yet, but it will break the first time someone tries to populate post→user.

## 4. Feature Status

### Fully working (traced through the code)
- User registration — hashes password with bcrypt, prevents duplicate usernames, sets JWT cookie.
- User login — verifies credentials, sets JWT cookie.
- JWT auth middleware — verifies token, loads `req.user`, blocks unauthenticated requests.
- End-to-end upload pipeline: image in → Gemini caption generated → image uploaded to ImageKit → API responds with caption + hosted URL.

### Partially built / buggy
- **Post persistence is broken** — see schema mismatch above; the `image` URL is never actually saved to MongoDB (caption and user ID are saved correctly).
- **No way to retrieve posts** — there's no GET endpoint, so even correctly-saved data is unreachable via the API today.
- **Insecure cookie config** — `res.cookie("token", token)` is set with no `httpOnly`, `secure`, or `sameSite` flags, meaning it's readable by client-side JS and sent over plain HTTP.
- **`jwt_secret` env var is undocumented** — the code reads `process.env.jwt_secret` (lowercase, non-standard naming) in both the controller and middleware, but the README's `.env` example never mentions this variable at all. If left unset, `jwt.sign`/`jwt.verify` will still "work" using the literal string `"undefined"` as the secret — a silent security hole rather than a startup crash.
- **`PORT` env var is inert** — README documents it, code never reads it (hardcoded to 3000).
- **`MONGODB_URI` vs `MONGO_URI` mismatch** — code reads `process.env.MONGODB_URI`; README's `.env` example tells you to set `MONGO_URI`. Following the README literally means DB connection fails on first run.

### Mentioned in docs/UI but not implemented
- Entire React/Tailwind frontend (README describes a full `client/` tree — zero files exist).
- "User profile" route (present only as a comment in `auth.routes.js`).
- Logout endpoint.
- "Multiple caption styles/tones," "caption history," "export options," "social platform integrations" — all listed under README's "Future Enhancements," none exist in code.

### External services required
- A MongoDB instance (local or Atlas)
- A Google AI Studio Gemini API key
- An ImageKit account (public key, private key, URL endpoint)

### Steps to run locally
1. `npm install` at the **repo root** (there is no `server/` or `client/` subfolder as README claims — `package.json` lives at the top level).
2. Create a `.env` file at the repo root with: `MONGODB_URI`, `jwt_secret`, `IMAGEKIT_PUBLIC_KEY`, `IMAGEKIT_PRIVATE_KEY`, `IMAGEKIT_URL_ENDPOINT`, `GEMINI_API_KEY`.
3. Run `node server.js` directly (or `npx nodemon server.js`) — **`package.json` has no `start`/`dev` script**, so `npm start` (as README instructs) will fail.

### Things that will block a first run
- README's `cd server` / `cd client` steps don't match the actual flat repo layout.
- `MONGO_URI` vs `MONGODB_URI` naming mismatch will silently break the DB connection.
- `jwt_secret` isn't documented anywhere in the README's setup instructions — auth will "work" but with a predictable/undefined signing secret unless you know to add it.
- `npm start` doesn't exist as a script.
- There is no frontend to run at all, despite step 3 of the README's setup guide.

## 6. Gap Analysis for Your Goal

**Goal:** AI brand-kit + social post generator — user stores brand guidelines (colors, fonts, logo, tone), types a prompt, gets on-brand social designs via LLM, exportable as PNG.

### Reusable directly
- Express app skeleton, Mongoose connection setup, JWT auth (register/login/middleware) — can become the account system as-is (after fixing the cookie/env issues above).
- Multer in-memory upload pattern — reusable for logo uploads.
- The Gemini SDK call structure in `ai.service.js` — reusable as the pattern for prompt-driven generation (though the model call itself needs to change — see below).
- ImageKit integration — reusable as the CDN/storage layer for exported PNGs and uploaded logos.

### Needs modification
- **`Post` model** → needs to become something like `BrandKit` (colors, fonts, logo asset, tone/voice description) plus a `GeneratedPost`/`Design` model (prompt, brand kit ref, rendered asset URL, platform/template).
- **`ai.service.js`** → currently does image-to-caption (vision input, text output). Your goal needs **text/brand-context → visual design output**, which is a different capability. `gemini-2.5-flash` does not generate images; you'll need either an actual image-generation model (Gemini's image-gen / Imagen, or another provider) or — the more controllable and genuinely "on-brand" approach — use the LLM only to produce structured design content (headline, subtext, layout choice) and render it onto a template using brand tokens via an HTML/CSS-to-image renderer (e.g., `satori` + `resvg`, or a headless-browser screenshot). This second approach gives far more reliable brand-color/font fidelity than asking a generative image model to "use these exact hex codes."
- Fix the `image`/`imageUrl` schema mismatch, the env var naming inconsistencies, and cookie security flags before building on top of the auth system.

### Needs to be built from scratch
- Brand kit CRUD (colors, fonts, logo, tone) and storage.
- Prompt-to-design generation service (as described above).
- A template/design system sized per platform (Instagram post/story, LinkedIn, X/Twitter card, etc.) with logo/color/font slots.
- PNG export/rendering pipeline.
- Any GET endpoints (list brand kits, list generated posts, fetch a single asset) — none exist today.
- The entire frontend — there is currently none.

### Prioritized Roadmap
